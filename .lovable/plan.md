

# Critical Fix: Custom CSS Layer Parse Failure Causing Section Collapse

## Problem

All homepage sections between Hero and CrossOver/Footer are collapsed to near-zero height. The page scrollHeight is approximately 2000px instead of 7500+. The orchestrator console logs confirm: progress jumps from 0.077 to 1.000 in a single scroll frame, meaning the process section's `rect.height` is effectively 0.

## Root Cause (Confirmed)

The `@layer utilities` block in `src/index.css` (lines 544-5408, approximately 4,865 lines) contains a CSS parse error that causes the browser to silently discard the ENTIRE block. This means:

- Tailwind's built-in utilities (`h-screen`, `flex`, `min-h-[70vh]`, `py-24`, etc.) still work because they are generated by `@tailwind utilities;` on line 3
- ALL custom utility classes fail: `.process-section` (min-height: 180vh), `.section--dark` (background color), `.section-padding-standard` (padding), `.exhale-section`, `.gradient-dawn`, etc.
- Inline `minHeight` styles were added but are insufficient because section content, backgrounds, and layout all depend on the broken custom classes
- The hero works because it uses only Tailwind built-in classes and inline styles
- CrossOver/Footer appear because they have inline background styles and enough Tailwind classes

## Fix Strategy: Move Critical Layout CSS Outside @layer

The safest fix is to extract the most critical layout-defining CSS rules OUT of `@layer utilities` and place them as plain CSS rules (outside any layer). This ensures they apply regardless of whether the layer block parses correctly. Simultaneously, add inline `minHeight` to ProcessSection (the only section that was missed).

### Step 1: Extract Critical Section CSS to Top-Level

Move these essential rules from inside `@layer utilities` to a new plain CSS block placed AFTER the `@layer utilities` closing brace. This guarantees they apply even if the layer block has parse issues:

```css
/* CRITICAL LAYOUT RULES — Outside @layer for resilience */
.process-section {
  position: relative;
  min-height: 180vh;
  padding: 120px 24px;
  display: flex;
  flex-direction: column;
  align-items: center;
  overflow: hidden;
}

.section--dark {
  background-color: hsl(var(--surface-dark-band));
  color: hsl(var(--foreground));
}

.section-padding-standard {
  padding-top: 80px;
  padding-bottom: 80px;
}

.section-padding-cinematic {
  padding-top: 120px;
  padding-bottom: 120px;
}

.exhale-section {
  /* Critical layout properties from the original definition */
}
```

### Step 2: Add Inline minHeight to ProcessSection

The ProcessSection.tsx was missed in the previous round. Add `style={{ minHeight: '180vh' }}` to the section element as a defensive backup.

### Step 3: Diagnose the Parse Error

Scan the `@layer utilities` block for common silent-failure causes:
- Unclosed comments (`/* ... ` without `*/`)
- Invalid `@apply` references to nonexistent Tailwind classes
- Stray characters or encoding issues
- Nested `@layer` blocks (which are invalid)
- Invalid CSS values (e.g., `hsl()` with missing commas in older syntax)

The most likely culprits are in the 4000+ lines of process section, held breath, letterpress card, and animation definitions where complex CSS was generated. A binary search approach (commenting out halves) can isolate the exact failing line.

### Step 4: Binary Search for Parse Error

To find the actual parse error, split the `@layer utilities` block into two halves using a temporary comment-out. If the first half works, the error is in the second half. Repeat until found. This is the only reliable way to find a silent CSS parse error in a 5000-line block.

| `src/components/process/ProcessSection.tsx` | Add `style={{ minHeight: '180vh' }}` to the section element |
| `src/index.css` | Binary search for and fix the parse error inside `@layer utilities` |

1. First: Add inline `minHeight: '180vh'` to ProcessSection.tsx (immediate fix for that section)
2. Second: Add critical layout rules as plain CSS after `@layer utilities` block (ensures all sections have backgrounds, padding, and min-heights regardless of layer parsing)
3. Third: Binary search for the parse error in the 5000-line block and fix it

## Expected Result

- All 13 sections render at proper heights
- Page scrollHeight returns to 7500px+
- Process orchestrator reports gradual progress (0.0 to 1.0 over scroll)
- Section backgrounds, padding, and typography apply correctly
- The fix is resilient: plain CSS fallbacks ensure sections never collapse again even if the layer block has issues
